## 二、逐项审核

### Bug #1 — `run.py` `case_count` NameError

**原问题：** Phase 2 使用未定义的变量 `case_count`，走到就直接 NameError 崩溃。

**修复方式：** 在 Phase 1 末尾加了 `case_count = parsed["case_count"]`，同时删了 Phase 2 残留的重复打印语句。

**审核结论：** ✅ 正确修复。直接从 parsed 字典取值，类型匹配，位置正确。

---

### Bug #2 — `boundary.py` `read_grouped()` 不存在

**原问题：** `use_grouped=True` 时调用不存在的方法，AttributeError。

**修复方式：** 从 `BoundarySummaryGenerator.generate()` 签名中删掉 `use_grouped` 参数，移除整个 `if/else` 分支，统一走 `read_all_sections()`。对应 CLI 入口 `generate_boundary.py` 的 `--grouped` 参数引用也同步移除。

**审核结论：** ✅ 正确修复。比较干脆——功能不完整直接砍掉而不是留个半成品。不影响主流程。

---

### Bug #3 — `api_client.py` fee=0 误判为成功

**原问题：** code=200 + fee=0 + 无 failureReason 时 success=True，报告列显示 PASS。

**修复方式：** 
- `else` 分支不再直接设 `success = True`
- 加了 `_debug_failure.json` 的失败日志输出
- failureReason 提取路径优先从 `info.failureReason` 取

**审核结论：** ✅ 正确修复。同时新增的失败日志落盘对调试 API 问题也很有用。

---

### Bug #4 — `rate_parser.py` 空年龄行截断

**原问题：** `_extract_section_data` 遇到 age=None 但费率列有值时 `break`，丢失该区段后续所有数据。

**修复方式：** `break` 改为 `continue`，并加了 `skipped_rows.append(row)` 记录跳过的行。

**审核结论：** ✅ 正确修复。只跳过当前行不影响后续数据，是合理的做法。加空的 `skipped_rows` 列表但没有进一步使用（没打日志或返回给调用方），可以用但留了个小尾巴。

---

### Bug #5 — `re.sub` 误截断责任名称

**原问题：** `re.sub(r'(责任|保险金责任|保险金)$', '', part)` 会把"重大疾病保险金"截成"重大疾病"。

**修复方式：** `web_app.py` 中改为 `re.sub(r'责任$', '', part)`，只去掉尾部"责任"二字。`rate_parser.py` 中的 `_extract_product_structure` 也有类似逻辑。

**审核结论：** ⚠️ 部分修复。`web_app.py` 里的改了，但 `rate_parser.py` 的 `_extract_product_structure` 也要确认是否同步。如果两个地方不一致（web 解析显示全名，CLI 解析还是截断的），会影响不同入口的一致性。

---

### Bug #6 — `File.path` 不可用

**原问题：** 前端用 `currentFile.path` 传报告保存路径，标准浏览器无此属性，报告存到不可预期的目录。

**修复方式：** 移除了对 `original_path` 的依赖，报告统一固定存到 `str(UPLOAD_DIR)`（即 `_uploads/` 目录）。

**审核结论：** ✅ 彻底修复。不再依赖浏览器端传路径，保存位置可预期。

---

### Bug #7 — `file://` 下载链接不可用

**原问题：** 前端用 `href="file://${data.report_path}"`，主流浏览器禁止从 HTTP 页面打开 file:// 链接。

**修复方式：** 新增 `/api/download` Flask 端点，通过 `send_file` 提供真实 HTTP 下载。前端改为 `download?task_id=xxx` 的方式。

**审核结论：** ✅ 正确修复。并且加了路径安全检查（`allowed_dirs`、`os.path.realpath`），比简单方案更稳妥。

---

### Bug #8 — `--account` CLI 参数被覆盖

**原问题：** 即使传了 `--account xxx`，只要没有 `--serial-no`，交互 `prompt_account()` 就覆盖掉 CLI 传入的值。

**修复方式：** 条件从 `if not args.serial_no:` 改为 `if not args.serial_no and not args.account:`。

**审核结论：** ✅ 正确修复。逻辑简单直接。

---

### Bug #9 — 登录失败后继续执行

**原问题：** 登录失败只打一行"⚠ 登录可能失败"就继续，所有用例因缺少 cookie 全 FAIL。

**修复方式：** 改为 `return []` + 清晰错误提示"❌ 登录失败，请检查账号密码配置"。

**审核结论：** ✅ 正确修复。可能唯一的小问题是应该在登录结果确信失败时不执行，但目前的启发式判断（`"成功" in resp.text`）还是有点松，容错和误判之间有个取舍。

---

### Bug #10 — `_tasks` 字典内存泄漏

**原问题：** SSE 完成/错误后从不删除 `_tasks` 记录，长期运行内存增长。

**修复方式：**
- 新增任务时检查并清理超过 10 分钟的 stale entries
- SSE complete 和 task_error 事件响应后执行 `_tasks.pop(task_id, None)`

**审核结论：** ✅ 正确修复。同时覆盖了"SSE 正常关闭"和"SSE 从未连接（stale）"两种情况。

---

### Bug #11 — 年龄列搜索上限 30 列

**原问题：** `max_col = min(ws.max_column, 30)` 限制了搜索范围。

**修复方式：** 去掉 cap，直接 `max_col = ws.max_column`。

**审核结论：** ✅ 已修复。同样的改动也应用到了 `_find_rate_columns` 里。

---

### Bug #12 — 年龄序列起始条件 ≤1 岁

**原问题：** `if a is not None and a <= 1` 导致纯成人产品（18 岁起）找不到年龄列。

**修复方式：** 重写为两阶段搜索：
1. 先尝试 `max_start_age=1`（找 ≤1 岁起始的序列）
2. 找不到则放宽到 `max_start_age=None`（接受任意起始年龄）

**审核结论：** ✅ 合理修复。两阶段策略比单条件更鲁棒，且优先匹配儿童产品常见格式。

---

### Bug #13 — data_type 互覆盖

**原问题：** `_read_metadata` 从产品信息 Sheet 读精确值，但随后 `_extract_metadata` 用粗糙正则覆盖。

**审核结论：** ❌ 未修复。两个来源的优先级逻辑保持原样。`_extract_metadata` 的 `data_type_code` 仍覆盖 `_read_metadata` 的 `data_type`。影响范围不大（"每X元保险费"这个正则误中率低），但确实是个残留的脆弱点。

---

### Bug #14 — 期望费率浮点精度

**原问题：** Excel 中存储的万分位费率经 openpyxl 读取可能有微小偏差。

**修复方式：** `_float()` 加了 `round(float(v), 10)`。

**审核结论：** ⚠️ 打了补丁但不是根本解决。`round(..., 10)` 能处理大多数情况，但如果费率位数超过 10 位小数（理论上保险行业不大会），或者 Excel 源数据偏差超过 10 位小数位，仍可能影响 PASS/FAIL 判定。建议配合 `Decimal` 使用会更彻底。

---

### Bug #15 — `ruitai.yaml` duty `"3"` default_value 类型不一致

**原问题：** duty "3" 的 `default_value` 是 `["3"]`（数组），其他 duty 是 `"3"`（字符串）。

**修复方式：** `["3"]` → `"3"`，`["4"]` → `"4"`。

**审核结论：** ✅ 已修复。同时修正了 legacy 目录下的对应配置文件。

## 三、额外修复评估

### 正面（提升质量）

| 改动 | 评估 |
|------|------|
| ThreadPoolExecutor 并行执行 | 从串行→20 线程并行，测算速度大幅提升。每个线程独立 client，login_cookies 共享，需确认 API 端能承受并发 |
| 金额限制智能重试 | 很实用的功能——API 返回"最低保额""步长"等限制时自动调整重试，减少人工介入 |
| SSE 实时进度 + 动态配置区 | Web 界面从简单的阻塞式请求改成了 SSE 推送，体验提升明显。动态展示责任码值、保险期间编码等产品配置 |
| 错误处理增强 | 文件保存、上传异常、500 的统一 JSON 返回，比原来 `threaded=True` 下的异常崩溃更健壮 |
| _debug_failure.json 日志 | 保存最近一次失败请求的 payload 和 response，对调试很实用 |

### 需要注意的

| 改动 | 潜在风险 |
|------|---------|
| 20 线程并发 | 默认 workers=20 会对 API 产生较大压力。如果 API 有频率限制或按量计费，可能需要调低 |
| 金额重试最多 10 次 | 极端情况下一个用例重复请求 11 次（1 次初始 + 10 次重试），可能慢且消耗 API 额度 |
| rate_parser.py 超大 diff | 1173 行变化相当于重写了大部分逻辑。代码行数从约 750 行膨胀到约 900 行，复杂度增加 |
