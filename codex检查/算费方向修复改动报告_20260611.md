## 一、修复的两个核心问题

### 问题 1：data_type 不参与计算逻辑

`tester.py` 的 `_run_single` 不管算费方向，全部按"保额算保费"处理。

**原来：**
```python
amount = random.randrange(...)
expected_premium = amount * expected_rate / 1000
# 然后调 API，比较返回的 fee 和 expected_premium
```

**如果上传的费率表是 type 2（保费算保额）**，这个计算就反了——输入应该是保费，期望值应该是保额，API 返回的也应该是保额而非保费。结果全是 FAIL，而且不知道为什么。

### 问题 2：费率单位硬编码 + 步长概念混淆

`fee_unit`（费率表里"每X元"中的 X）硬编码为 1000，和 `test.amount.step`（产品实际的保额/保费步长）不是同一个东西。

```
费率表：每 1,000 元保额 → 保费 4.20 元
产品限制：每 10,000 元增减保额
```

原来两个都硬编码/混用，数据流断了。

## 二、改动详解

### 1. src/config.py — 新增 data_type/fee_unit 字段

```python
class ProductProfile:
    def __init__(self, data: dict):
        # 新增
        self.data_type: str = str(data.get("data_type", "1"))  # "1" 或 "2"
        self.fee_unit: int = int(data.get("fee_unit", 1000))

    # 新增快捷判断
    @property
    def is_premium_to_coverage(self) -> bool:
        return self.data_type == "2"
```

数据流起点，让 ProductProfile 能承载算费方向和费率单位。

### 3. src/api_client.py — save_product 支持双向

```python
def save_product(
    self,
    plan, ensure_period, pay_period,
    amount: int = 0,           # ← 不再强制必填，默认0
    ensure_plan="1",
    duty_list=None, cookies=None,
    input_type: str = "amount",  # ← 新增参数
) -> tuple:
```

**payload 构建方向感知：**
```python
if input_type == "amount":
    # 保额算保费：发保额，等保费回来
    amount_descr, amount_val, fee_val = str(amount), str(amount), ""
else:
    # 保费算保额：发保费，等保额回来
    amount_descr, amount_val, fee_val = "", "", str(amount)
```

**响应提取方向感知：**
```python
if input_type == "amount":
    result_value = self._extract_fee(result)            # 读 fee 字段
else:
    result_value = self._extract_sum_insured(result)    # 读 amount 字段
```

新增 `_extract_sum_insured()` 方法与 `_extract_fee()` 对称，从相同的三层路径（result.info → proposalPlanVOList → result 顶层）提取 amount 字段。

### 4. src/tester.py — _run_single 完全重写

这是最关键的改动。`_run_single` 的核心逻辑变成了：

```python
data_type = self.profile.data_type
fee_unit = self.profile.fee_unit or 1000
is_type2 = (data_type == "2")
input_type = "premium" if is_type2 else "amount"

# 生成输入值（统一步长）
if test_cfg.fixed:
    input_val = test_cfg.fixed
else:
    input_val = random.randrange(test_cfg.min, test_cfg.max + 1, test_cfg.step)

# 期望值计算（方向感知）
if is_type2:
    # 保费算保额：期望保额 = 保费 × 费率单位 ÷ 费率
    expected_output = input_val * fee_unit / expected_rate
    input_label = "保费(元)"
    output_label = "期望保额(元)"
else:
    # 保额算保费：期望保费 = 保额 × 费率 ÷ 费率单位
    expected_output = input_val * expected_rate / fee_unit
    input_label = "保额(元)"
    output_label = "期望保费(元)"
```

**结果字典动态列名：**
```python
result = {
    ...
    input_label: input_val,          # "保额(元)" 或 "保费(元)"
    output_label: round(expected_output, 2),  # "期望保费(元)" 或 "期望保额(元)"
    "API返回值": None,               # 改成通用名，不再叫 "API返回fee"
    "_data_type": data_type,         # 内部标记，给 reporter 用
    "_input_label": input_label,
    "_output_label": output_label,
}
```

**API 调用传 input_type：**
```python
ok, result_val, status, text, reason = api.save_product(
    ...,
    amount=input_val,
    input_type=input_type,  # ← 告诉 API 客户端方向
)
```

**重试日志也改：**
```python
retry_log.append(f"...{'保费' if is_type2 else '保额'} {old_val:,} → {input_val:,}...")
```

## 四、改后比改前的变化

| 项目 | 改前 | 改后 |
|------|------|------|
| data_type 使用 | 只显示在页面预览里，不参与计算 | 全程参与：输入生成、期望计算、API 调用、报告列名 |
| 期望值计算 | `amount × rate / 1000`（硬编码） | `amount × rate / fee_unit`（type1）<br>`premium × fee_unit / rate`（type2） |
| API payload | 始终发 amount | 根据 input_type 发 amount 或 fee |
| 响应提取 | 始终提取 fee | 根据 input_type 提取 fee 或 amount |
| 报告列名 | 固定"保额(元)""期望保费(元)""API返回fee" | `"算费方向":"保额算保费"`<br>`"保额(元)"` 或 `"保费(元)"`<br>`"期望保费(元)"` 或 `"期望保额(元)"`<br>`"API返回值"` |
| 列数 | 固定 24 列 | 动态（当前方向允许的列 + 结果中存在的键） |

## 五、边界情况说明

- **默认值回退**：如果解析结果中没有 `data_type_code`，默认按 type 1（保额算保费）处理，不影响旧费率表
- **费率单位缺失**：如果解析没有提取到 `fee_unit`，默认 1000，与之前行为一致
- **run_tests.py 未改**：格式驱动路径（`run_tests.py`）不在本次改动范围内。如果需要，后续可以加——原理一样，把 `RateTableMetadata` 里的 data_type/fee_unit 注入到 `ProductProfile` 即可
- **run.py 的 parsd["data_type"] 中文标签**：这个字段名字没变，只是新增了 `parsed["data_type_code"]`，前端预览不受影响
